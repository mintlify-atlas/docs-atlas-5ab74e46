## Core Principles

### Why Database Agnosticism?

CBDB Online maintains database-agnostic code for several strategic reasons:

<AccordionGroup>
  <Accordion title="Future Migration Flexibility">
    The project may need to migrate to PostgreSQL, MySQL 8.x, or other database systems in the future. Avoiding vendor lock-in ensures smooth transitions.
  </Accordion>
  
  <Accordion title="Development Environment Diversity">
    Developers may use different database versions or systems. Database-agnostic code ensures consistency across environments.
  </Accordion>
  
  <Accordion title="CI/CD Efficiency">
    Test environments use SQLite in-memory databases for speed. Code must work across both MySQL/MariaDB and SQLite.
  </Accordion>
  
  <Accordion title="Cost Considerations">
    Avoiding vendor-specific commercial features reduces dependency on specific database vendors.
  </Accordion>
</AccordionGroup>

### Recommended Practices

<CardGroup cols={2}>
  <Card title="Use Standard SQL" icon="check">
    ```sql
    -- ✅ Good: Standard SQL
    SELECT * FROM users 
    WHERE name LIKE 'John%';
    
    -- ✅ Good: Cross-database indexes
    CREATE INDEX idx_name ON users(name);
    ```
  </Card>
  
  <Card title="Use Laravel Query Builder" icon="check">
    ```php
    // ✅ Good: Laravel abstraction layer
    DB::table('users')
        ->where('name', 'like', 'John%')
        ->get();
    ```
  </Card>
</CardGroup>

<CardGroup cols={2}>
  <Card title="Use B-Tree Indexes" icon="check">
    Supported by all database systems
    
    - Single column indexes
    - UNIQUE indexes
    - Composite indexes
  </Card>
  
  <Card title="Avoid Vendor-Specific Features" icon="x">
    ```sql
    -- ❌ Bad: MySQL-specific ngram parser
    CREATE FULLTEXT INDEX idx_name
    ON table_name(column_name) 
    WITH PARSER ngram;
    
    -- ❌ Bad: MariaDB-specific plugin
    INSTALL SONAME 'ha_spider';
    ```
  </Card>
</CardGroup>

### Compatibility Guidelines

<Tabs>
  <Tab title="✅ Use">
    - Laravel Schema Builder
    - Standard SQL syntax
    - B-Tree indexes (default)
    - Application-layer full-text search (Elasticsearch, Meilisearch)
    - Standard LIKE patterns for prefix matching
  </Tab>
  
  <Tab title="❌ Avoid">
    - Database-specific functions (REGEXP, optimizer hints)
    - Vendor-specific syntax
    - Full-text search with database-specific parsers
    - Generated columns
    - Database-specific storage engines
  </Tab>
</Tabs>

## Index Strategy

### B-Tree Indexes (Recommended)

<Card title="Benefits" icon="bolt">
  - ✅ Supported by all databases
  - ✅ Excellent for prefix matching (`LIKE 'prefix%'`)
  - ✅ Efficient for sorting and range queries
  - ✅ Supports composite indexes
</Card>

```php
Schema::table('users', function (Blueprint $table) {
    $table->index('name');                      // Single column
    $table->index(['last_name', 'first_name']); // Composite
    $table->unique('email');                    // Unique
});
```

### Composite Index Rules

<Warning>
  Queries must include the **leading columns** of the index to use it effectively.
</Warning>

**Example**: Index on `['last_name', 'first_name']`

- ✅ **Can use index**: `WHERE last_name = 'Smith'`
- ✅ **Can use index**: `WHERE last_name = 'Smith' AND first_name = 'John'`
- ❌ **Cannot use index**: `WHERE first_name = 'John'` (skips leading column)

## ORM Strategy

<Tip>
  **Important**: Laravel Eloquent ORM officially does not support composite primary keys. Choose the appropriate data access method based on table structure.
</Tip>

<Tabs>
  <Tab title="Single Primary Key">
    ```php
    // ✅ Good: Use Eloquent for single primary key tables
    class User extends Model {
        protected $primaryKey = 'id';
    }
    
    $user = User::find(1);
    $user->update(['name' => 'New Name']);
    $user->delete();
    ```
  </Tab>
  
  <Tab title="Composite Primary Key">
    ```php
    // ✅ Good: Use Query Builder for composite primary key tables
    // Example: ALTNAME_DATA has composite key 
    // (c_personid, c_sequence, c_alt_name_chn, c_alt_name_type_code)
    
    DB::table('ALTNAME_DATA')
        ->where('c_personid', $personId)
        ->where('c_sequence', $sequence)
        ->update(['c_alt_name_chn' => $name]);
    
    DB::table('ALTNAME_DATA')
        ->where('c_personid', $personId)
        ->where('c_sequence', $sequence)
        ->delete();
    ```
    
    <Warning>
      **Do not create Eloquent models for composite primary key tables**. Eloquent's `delete()`, `update()`, and `find()` methods will not work correctly.
    </Warning>
  </Tab>
</Tabs>

### Handling Side Effects

For tables using Query Builder (composite keys), handle side effects explicitly:

```php
// Don't rely on Eloquent Observers (they won't fire for Query Builder)
// Instead, manually call related services in Repository or Service layer

// Example: Update name and refresh search index
DB::table('ALTNAME_DATA')->where(...)->update(...);
app(NameSearchIndexService::class)->refreshForPerson($personId);
```

<Note>
  This approach is more explicit and easier to test than relying on model observers.
</Note>
